# Story 8 포트폴리오 작성 Task 계획

이 문서는 Epic 2 결과물(React SPA + Swagger)을 프리랜서 클라이언트 시선의 포트폴리오로 정리하는 작업을 분해한다. 코딩이 아니라 문서 작성 작업이므로 완료 단위는 "검토 단위"다.

---

## 1. 왜 두 Task로 나누는가

포트폴리오 작성은 두 가지 다른 인지적 작업으로 구성된다.

- **Task 1 (초안)**: 내가 만든 것을 안에서부터 정리한다. 무엇을 만들었고, 왜 그렇게 골랐고, Epic 1과 어떻게 이어지는지를 자기 시선으로 글로 옮기는 단계
- **Task 2 (외부 시선 검수)**: 그 글을 "나를 모르는 클라이언트"의 시선으로 다시 읽고 다듬는다. 자기 시선으로는 보이지 않던 비약·전제·전문용어 과다 사용을 잡는 단계

두 작업을 한 번에 시도하면 보통 초안 정리에 끌려가서 검수가 약해진다. 글이 머리에 익으면 외부 시선이 점점 어려워진다. 초안을 한 번 검토받고 잠시 거리를 둔 뒤 검수로 넘어가는 것이 검수 품질을 지키는 방법이다.

---

## 2. 고정 결정

- 저장 위치: `problems/order-platform/order-platform-management/portfolio/epic2.md` (Epic 1 포트폴리오와 같은 디렉터리에 두어 연결성 확보)
- 분량: 한 문서. 별도 시리즈·블로그 형식 미도입
- 독자: 프리랜서 클라이언트. 동료 엔지니어 시선이 아니라 "이 사람을 고용할지" 판단해야 하는 사람
- 형식: Markdown. 코드 스니펫·다이어그램은 필요 시 추가하되 의무 아님
- 시점: B 완료 직후(Story 7 머지 후) 작성. 결정 맥락이 생생할 때 한 번에 마무리하는 것이 이 Story의 전제

---

## 3. 권장 Task 순서

### Task 1. 포트폴리오 초안 작성 (자기 시선)

#### 목표

Epic 2에서 무엇을 만들었고, 왜 그 기술 스택을 골랐고, Epic 1과 어떻게 이어지는지를 한 문서로 정리한 초안을 만든다. 클라이언트 시선 다듬기는 다음 Task로 넘긴다.

#### 핵심 작업

문서가 다음 4개 블록을 담는다.

- **무엇을 만들었는가**
  - React SPA + Swagger UI가 한 흐름으로 동작하는 결과물
  - 주문 생성 → 자동 이동 → 상태 이력 타임라인까지의 사용자 흐름
  - 핵심 화면 캡처 또는 화면 설명 (캡처는 가능하면 첨부)
- **왜 이 기술 스택인가**
  - Vite + React + TypeScript 선택 근거 (Story 3 ADR 산출물 참조)
  - 상태 관리 도구 선택 근거 (Story 4 ADR 산출물 참조)
  - Swagger 어노테이션 정책의 의도 (단순 노출이 아니라 "명세서 수준"으로 가져간 이유)
- **어떻게 구조를 잡았는가**
  - 3계층(View / ViewModel / API) 분리의 의도
  - AOP로 이력 삽입을 도메인 서비스에서 분리한 이유
- **Epic 1과 어떻게 이어지는가**
  - "백엔드 Saga만 있던 프로젝트 위에 풀스택 데모를 쌓은 흐름"
  - Epic 1 포트폴리오에서 끝난 지점이 Epic 2에서 어떻게 이어졌는지 한두 문단

#### 설계 포인트

- ADR 산출물(`epic2/adr/`)과 Story별 산출물에서 결정 이유를 가져온다. 새로 발명하지 않음
- 한 블록당 1~3 문단을 넘기지 않게 분량을 압축. 클라이언트는 길이가 아니라 명확함을 본다
- 시간 순서가 아니라 "독자가 알고 싶어할 순서"로 배치 — "무엇을 만들었나"가 가장 위, "기술 디테일"이 가장 아래

#### 이 검토 단위에서 하지 않을 것

- 외부 시선 다듬기·문장 압축 (Task 2)
- 이력서·자기소개 톤 정리 — Epic 8(가칭 포트폴리오 종합) 범위로 미룸
- Epic 1 포트폴리오 자체를 손보는 작업 — 이 Task는 Epic 2만

#### 완료 조건 (검토 가능한 상태)

- 위 4개 블록이 모두 채워진 초안 파일이 `portfolio/epic2.md`에 존재
- 사용자가 한 번 통독하고 "내가 만든 것이 빠짐없이 들어갔다"고 판단할 수 있는 상태
- 다음 Task에서 다듬을 항목(예: "여기 너무 전문적임", "이 문단이 비약")이 메모로 남아 있음 (검수 입력값)

#### 사용자 직접 확인 포인트

- 초안 통독 후, "내가 Epic 2에서 결정한 것 중 빠진 것이 있는가" 점검
- 다음 Task에서 외부 시선으로 보완할 후보 목록을 메모

---

### Task 2. 외부 시선 검수 + 다듬기

#### 목표

초안을 "나를 모르는 클라이언트"의 시선으로 다시 읽고, 비약·전문용어 과다·근거 부족을 잡아 최종본으로 다듬는다. 이 Task가 끝나면 그대로 외부에 제출 가능한 상태가 된다.

#### 핵심 작업

- 외부 시선 통독
  - "이 사람이 풀스택을 할 수 있는가?"라는 질문을 머리에 두고 처음부터 끝까지 읽기
  - 전문용어가 등장한 위치마다 "독자가 이 단어를 알고 있다고 가정해도 되는가" 점검
  - 결정 근거가 약한 곳(예: "Vite를 선택했다"만 있고 비교 대상이 없음) 찾기
- 다듬기
  - Task 1에서 메모한 보완 항목 + 통독으로 새로 발견된 항목 모두 반영
  - 비약 줄이기, 단락 순서 재배치, 불필요한 디테일 제거
- Epic 1 포트폴리오와의 연결 점검
  - 두 문서를 연달아 읽었을 때 흐름이 자연스러운지 확인
  - Epic 1에서 끝난 지점과 Epic 2에서 시작한 지점이 같은 언어로 묘사되는지 확인

#### 설계 포인트

- 새 내용 추가보다는 **삭제·재배치**를 우선. 클라이언트용 글의 핵심은 분량 압축
- "이 문단이 없어도 의미가 통하는가"를 각 문단마다 점검

#### 이 검토 단위에서 하지 않을 것

- 새 기술 결정·새 산출물 추가 — Story 1~7 범위가 아닌 항목은 포트폴리오에 등장시키지 않음
- 디자인·CSS 다듬기 — 문서 자체가 대상이지 결과물 화면이 대상이 아님
- 다른 Epic의 포트폴리오 손보기

#### 완료 조건 (검토 가능한 상태)

- 사용자가 다음 자가 평가 질문에 모두 "예"로 답할 수 있는 상태
  - [ ] 내가 모르는 클라이언트가 이 문서만 읽고 나를 고용할 수 있는가
  - [ ] Epic 1 포트폴리오와 연달아 읽었을 때 흐름이 자연스러운가
  - [ ] 전문용어가 처음 등장한 자리에 한 줄 설명이 있거나, 없어도 문맥으로 이해되는가
  - [ ] 각 기술 선택에 대해 "왜 그것인가"가 한 문장으로 답해지는가
  - [ ] 분량이 클라이언트가 한 번에 통독할 수 있는 수준인가

#### 사용자 직접 확인 포인트

- 자가 평가 질문 전부 통과 → 머지
- 통과하지 못하면 막힌 항목만 추가 다듬기 후 재검토 (Task 2 안에서 사이클 반복)

---

## 4. 이 계획에서 의도적으로 뒤로 미룬 것

- 다른 Epic의 포트폴리오 작성 — 각 Epic이 끝난 직후에 작성
- 전체 포트폴리오 종합 문서(이력서/자기소개 톤) — 별도 Epic 또는 별도 문제 정의 단계
- 외부 검수자(타인)에게 보여주고 피드백 받기 — 필요 시 Task 2 이후 별도 사이클로 진행
- 발표용 슬라이드·동영상 변환 — 포맷 전환 작업은 본 Story 범위 밖

---

## 5. 권장 순서 요약

1. ADR 산출물과 Story별 산출물을 다시 훑어 결정 이유 수집
2. 4개 블록 구조로 초안 작성 → Task 1 검토
3. (잠시 거리 두기 — 가능하면 한나절 이상)
4. 외부 시선으로 통독 → 보완 항목 메모
5. 삭제·재배치·문장 다듬기
6. 자가 평가 질문 5개 모두 통과 → Task 2 머지
