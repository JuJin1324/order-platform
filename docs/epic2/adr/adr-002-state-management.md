# ADR-002: 상태 관리 도구 선택

## 상태

채택 (Accepted)

## 맥락

React SPA에서 컴포넌트 간 상태를 어떻게 공유하고 관리할지 결정해야 한다. Epic 2의 화면은 주문 생성 페이지와 주문 상태 조회 페이지, 두 개다. 두 페이지 사이에서 주고받아야 하는 상태는 주문 생성 폼 입력값, API 응답으로 받은 주문 ID, 상태 조회 결과 — 세 가지가 전부다.

## 후보군

### 1. Redux

전역 상태 저장소를 두고 모든 상태 변경을 action → reducer 흐름으로 관리한다.

- **장점:** 상태 변화 추적이 용이하다. 여러 컴포넌트가 동일한 상태를 공유할 때 props drilling 없이 접근할 수 있다.
- **단점:** 보일러플레이트가 많고 학습 곡선이 있다. 상태 규모가 작은 경우 오히려 구조가 복잡해진다. React 자체를 배우는 데 써야 할 주의가 분산된다.

### 2. Context API

React 내장 기능으로 컴포넌트 트리 전체에 상태를 공유한다.

- **장점:** 외부 라이브러리 없이 사용 가능하다. Redux보다 보일러플레이트가 적다.
- **단점:** 상태가 자주 바뀌면 불필요한 리렌더링이 발생할 수 있다. 상태 추적 도구가 없어 디버깅이 불편하다.

### 3. useState + React Router navigate(state)

각 컴포넌트가 로컬 상태를 가지고, 페이지 간 데이터는 React Router의 `navigate(state)`로 넘긴다.

- **장점:** 추가 라이브러리 없이 React 기본 기능만 사용한다. 상태 규모에 정확히 맞는 복잡도를 가진다.
- **단점:** 상태 공유 범위가 넓어지면 한계가 드러난다. props drilling이 깊어지면 관리가 어려워진다.

## 결정

**useState + React Router navigate(state)를 선택한다.**

## 근거

Redux나 Context API가 필요해지는 시점은 여러 컴포넌트가 같은 상태를 공유해야 하거나, 상태 전환이 복잡해지거나, props drilling이 고통스러워질 때다. Epic 2는 이 중 어느 것도 해당하지 않는다.

페이지가 2개이고 두 페이지 사이에 주문 ID 하나만 넘기면 된다. `useState` + `navigate(state)`로 충분히 해결된다. 지금 Redux를 도입하면 React 자체를 배우는 데 써야 할 주의가 분산된다. 상태 공유 필요성이 실제로 생길 때 Context API 또는 Redux로 자연스럽게 전환하는 것이 올바른 순서다.
