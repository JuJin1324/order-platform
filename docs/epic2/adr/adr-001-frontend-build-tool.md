# ADR-001: 프론트엔드 빌드 도구 선택

## 상태

채택 (Accepted)

## 맥락

Epic 1에서 Spring Boot 백엔드가 완성된 상태에서 풀스택 포트폴리오용 프론트엔드를 추가한다. 이 프로젝트에서 프론트엔드의 역할은 두 가지다: 백엔드 API를 UI로 연결하는 것, 그리고 SPA가 무엇인지 직접 학습하는 것. 빌드 도구 선택은 이 두 목적 모두에 영향을 미친다.

## 후보군

### 1. Vite + React + TypeScript

현재 SPA 개발의 표준 조합. Vite는 빌드 도구에 집중하고, 서버 기능은 제공하지 않는다.

- **장점:** Spring Boot와 역할이 완전히 분리된다. SPA 개념 자체에만 집중할 수 있어 학습 목적과 직접 일치한다. `react-ts` 템플릿을 기본 제공하므로 TypeScript 설정 비용이 없다.
- **단점:** SSR이 필요한 요구사항이 생기면 별도 대응이 필요하다.

### 2. Next.js

React 기반 풀스택 프레임워크. SSR, 파일 기반 라우팅, API Routes 등을 내장한다.

- **장점:** SSR·SEO가 필요한 프로덕션 서비스에 적합하다. 하나의 프레임워크로 프론트·백엔드를 모두 다룰 수 있다.
- **단점:** 이미 Spring Boot 백엔드가 있으므로 자체 서버 기능이 역할 중복이 된다. SSR·파일 기반 라우팅 같은 추가 개념을 함께 배워야 해서 SPA 학습 목적이 분산된다.

### 3. Create React App (CRA)

과거 React SPA의 표준 도구였으나 현재 유지보수가 중단됐다.

- **장점:** 학습 자료가 풍부하고 구조가 단순하다.
- **단점:** 2023년 이후 공식적으로 deprecated 처리됐다. 신규 프로젝트에 사용하는 것은 적절하지 않다.

## 결정

**Vite + React + TypeScript를 선택한다.**

## 근거

Spring Boot 백엔드가 이미 있으므로 Next.js의 서버 기능은 오버엔지니어링이다. "백엔드는 Spring Boot, 프론트엔드는 SPA"라는 구조를 명확히 유지하는 데 Vite + React가 가장 적합하다. CRA는 유지보수가 중단됐으므로 선택지에서 제외된다.

TypeScript는 별도 학습 비용 없이 API 응답 타입(`OrderResponse`, `OrderStatus` 등)을 명시적으로 정의할 수 있어, 백엔드 스펙 변경 시 프론트엔드 영향 범위가 즉시 드러난다는 이점이 있다.
