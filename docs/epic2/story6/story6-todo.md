# Story 6 TODO — 긴급 구현에서 의도적으로 미룬 것들

Story 6은 "Saga 흐름이 UI에 보인다"는 최소 기준으로 빠르게 구현한다. 아래 항목들은 현재 구현의 한계와 차후 개선 포인트다.

---

## 백엔드

### 1. OrderStatusHistory 트랜잭션 경계 ✅ Task 1 반영

**현재 결정**: 이력 삽입을 `confirmOrder`, `cancelOrder` 등 도메인 로직과 같은 트랜잭션에서 처리한다.

**문제**: 이력 삽입 실패가 주문 상태 전이 자체를 롤백시킬 수 있다. 감사 로그가 핵심 비즈니스 로직과 운명을 같이 하는 구조다.

**대안**: order-service 내부 도메인 이벤트를 발행하고, 별도 트랜잭션에서 이력을 기록하는 방식. 이력 삽입 실패가 주문 처리에 영향을 주지 않는다.

**참고**: Kafka 이벤트를 구독하는 방식도 있지만, 이는 서비스 간 결합을 추가하고 이력의 시간 정확도 문제가 생긴다(Kafka 컨슈머 처리 시점 ≠ 상태 전이 시점).

---

### 2. 단위 테스트 ✅ Task 1 반영

**현재 결정**: 기존 전체 테스트 통과 + 신규 통합 테스트 추가.

**부족한 것**:
- `OrderStatusHistory` 삽입 로직에 대한 단위 테스트
- 이력 삽입 실패 시 롤백 동작 검증
- `GET /orders/{id}/status-history` 컨트롤러 단위 테스트

---

### 3. 이력 삽입 방식 ✅ Task 1 반영 (AOP 선택)

**현재 결정**: 도메인 서비스에서 `OrderStatusHistoryRepository`를 직접 호출한다.

**문제**: 도메인 서비스가 이력 저장 책임까지 알게 된다. 이력 기록 방식이 바뀔 때 도메인 서비스를 수정해야 한다.

**대안**:
- Spring AOP로 상태 전이 메서드를 가로채 이력 삽입
- 도메인 이벤트(`OrderStatusChangedEvent`)를 발행하고 이벤트 리스너가 이력 삽입

---

### 4. `orderId` 컬럼 인덱싱 ✅ Task 1 반영

`OrderStatusHistory` 테이블에서 `orderId`로 조회하는 쿼리가 자주 실행된다. 데이터가 쌓이면 풀 스캔이 발생할 수 있다. `orderId`에 인덱스 추가 필요.

---

### 5. `changedAt` 시간 정확도

이력의 `changedAt`은 이력 레코드가 삽입되는 시점(`LocalDateTime.now()`)으로 기록된다. Kafka 컨슈머가 이벤트를 처리하는 시점은 이벤트 발행 시점보다 늦을 수 있어, 실제 상태 전이 시점과 `changedAt`이 다를 수 있다.

정확한 시점이 필요하다면 Kafka 이벤트 페이로드에 상태 전이 시각을 포함시켜야 한다.

---

### 6. 기존 주문 이력 정합성

서비스 재배포 후 이미 `CONFIRMED` 또는 `CANCELLED` 상태인 주문들은 `OrderStatusHistory`에 이력이 없다. 이 주문들을 조회하면 이력이 비어 있는 것처럼 보인다.

운영 환경이라면 마이그레이션 스크립트로 기존 주문의 현재 상태를 이력에 채워야 한다.

---

## 프론트엔드

### 7. `useOrderStatus` Hook 재사용성

현재 `useOrderStatus`는 이력 조회(`getOrderStatusHistory`)만 담당한다. 향후 현재 주문 상태(`getOrder`)도 같이 보여주는 요구사항이 생기면 Hook 분리 또는 통합 결정이 필요하다.

---

### 8. 타임라인 시간 표시 형식

`changedAt`은 ISO 8601 형식으로 내려온다. 현재는 raw 문자열을 그대로 표시한다. 사용자 친화적인 형식(`2026-04-22 14:30:05` 등)으로 포맷팅 필요.

---

### 9. 자동 갱신

현재는 수동 새로고침만 지원한다. Saga 흐름 관찰 경험을 위해 의도적으로 선택했지만, 실제 사용성을 위해서는 polling 또는 SSE 도입이 필요하다.
