# ADR-001: MSA 부분 커밋 불일치 해결을 위한 아키텍처 선택

## 상태

채택 (Accepted)

## 맥락

주문·결제·재고 세 서비스가 각자의 DB를 가진 MSA 구조에서, 중간 단계가 실패하면 이미 커밋된 앞 단계를 되돌릴 수단이 없다. REST 호출 체이닝으로 이 문제를 직접 재현했고(`주문=실패, 결제=성공`), 이 불일치를 해결할 아키텍처를 선택해야 했다.

## 후보군

### 1. 공유 DB

모든 서비스가 하나의 DB를 사용하면 단일 트랜잭션으로 원자성을 확보할 수 있다.

- **장점:** 구현이 단순하고, 기존 DB 트랜잭션만으로 일관성 보장
- **단점:** MSA의 전제(서비스별 독립 DB, 독립 배포, 독립 스케일링)를 포기해야 한다. 서비스 간 스키마 결합이 생기면 독립적 변경이 불가능해진다.

### 2. 2PC (Two-Phase Commit)

분산 트랜잭션 코디네이터가 모든 참여 서비스에 prepare → commit 순서로 원자적 커밋을 조율한다.

- **장점:** 강한 일관성(Strong Consistency) 확보
- **단점:** 코디네이터가 SPOF가 되고, prepare 단계에서 모든 참여 서비스가 락을 잡고 대기하므로 성능·가용성이 저하된다. 참여 서비스가 늘어날수록 실패 확률과 지연이 함께 증가한다.

### 3. REST + 보상 API

REST 동기 호출을 유지하되, 각 서비스에 취소 API를 두고 호출자가 보상 순서를 관리한다.

- **장점:** 기존 REST 통신 구조를 유지할 수 있어 진입 비용이 낮다
- **단점:** 동기 호출 체인의 결합도가 그대로 남는다. 보상 조율 책임이 호출자에 집중되어, 서비스가 추가되거나 보상 경로가 복잡해지면 호출자의 분기 로직이 비대해진다.

### 4. EDA + Saga

서비스 간 통신을 이벤트로 전환하고, 실패 시 보상 이벤트 체인으로 이미 커밋된 상태를 되돌린다.

- **장점:** 서비스 간 직접 의존을 제거한다. 각 서비스는 이벤트 계약만 알면 되고, 새 서비스 추가 시 기존 서비스를 수정할 필요가 없다.
- **단점:** 즉시 일관성 대신 결과적 일관성(Eventual Consistency)을 수용해야 한다. 전체 흐름을 한 곳에서 추적하기 어렵고, 이벤트 순서·중복·유실에 대한 추가 설계가 필요하다.

## 결정

**EDA + Saga를 선택한다.**

## 근거

이 프로젝트의 전제는 서비스별 독립 DB를 가진 MSA 구조다. 공유 DB는 이 전제 자체를 무효화한다. 2PC는 강한 일관성을 제공하지만, 코디네이터 의존과 락 대기로 인해 서비스가 늘어날수록 가용성이 떨어진다.

남은 선택지는 REST + 보상 API와 EDA + Saga다. 둘 다 보상 기반으로 결과적 일관성을 추구하지만, REST + 보상 API는 동기 호출 체인의 결합도를 그대로 유지한다. 호출자가 각 서비스의 보상 API를 직접 알고 순서대로 호출해야 하므로, 서비스 간 의존이 남는다. EDA에서는 각 서비스가 이벤트에만 반응하므로 이 직접 의존이 제거된다.

결과적 일관성 수용과 흐름 추적 복잡도 증가라는 트레이드오프가 있지만, 서비스 간 직접 의존 제거가 이 프로젝트에서 더 중요한 설계 목표라고 판단했다.
