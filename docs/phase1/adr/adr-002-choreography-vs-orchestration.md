# ADR-002: Saga 보상 패턴 선택 — Choreography vs Orchestration

## 상태

채택 (Accepted)

## 맥락

ADR-001에서 EDA + Saga를 선택했다. 다음으로 보상 흐름을 어떻게 조율할지 결정해야 한다. Saga의 보상 패턴은 크게 Choreography와 Orchestration 두 가지다.

## 후보군

### 1. Orchestration

중앙 Orchestrator가 각 서비스에 명령을 보내고, 응답에 따라 다음 단계를 결정한다. 실패 시 Orchestrator가 보상 순서를 직접 조율한다.

- **장점:** 전체 흐름이 Orchestrator 한 곳에 집중되어 있어 파악·디버깅이 쉽다. 보상 경로가 복잡해져도 Orchestrator 내부 로직만 수정하면 된다.
- **단점:** 모든 서비스가 Orchestrator에 결합된다. Orchestrator가 각 서비스의 명령과 응답을 알아야 하므로, 서비스가 추가되거나 변경될 때 Orchestrator도 함께 수정해야 한다.

### 2. Choreography

중앙 조율자 없이, 각 서비스가 자신이 수신한 이벤트에만 반응해 다음 이벤트를 발행한다. 실패 시에도 보상 이벤트가 역방향으로 전파되며 각 서비스가 자체적으로 보상을 수행한다.

- **장점:** 각 서비스는 이벤트 계약만 알면 된다. 서비스 간 직접 의존이 없으므로 새 서비스 추가 시 기존 서비스를 수정할 필요가 없다.
- **단점:** 전체 흐름을 한 곳에서 추적하기 어렵다. 서비스 수가 늘어나면 이벤트 체인이 복잡해지고, 흐름 파악을 위해 별도 수단(이벤트 스토밍 다이어그램, 분산 추적 등)이 필요하다.

## 결정

**Choreography를 선택한다.**

## 근거

ADR-001에서 EDA를 선택한 핵심 이유는 서비스 간 직접 의존 제거였다. Orchestration은 중앙 조율자를 통해 이 의존을 다시 만들어낸다. 모든 서비스가 Orchestrator에 결합되면, EDA를 선택한 근거와 충돌한다.

Choreography의 트레이드오프인 흐름 추적 어려움은 이 프로젝트의 규모(세 서비스, 순방향·역방향 각 1개 경로)에서는 이벤트 스토밍 다이어그램으로 충분히 대응할 수 있다고 판단했다. 서비스 수가 크게 늘어나 이벤트 체인이 복잡해지는 시점이 오면 Orchestration 재검토가 필요하다.
