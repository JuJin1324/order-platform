# ADR-003: 메시징 시스템 선택 — Kafka vs RabbitMQ vs SQS

## 상태

채택 (Accepted)

## 맥락

ADR-001에서 EDA를, ADR-002에서 Choreography Saga를 선택했다. 각 서비스가 이벤트를 발행하고 수신하는 구조를 구현할 메시징 시스템을 선택해야 한다.

Choreography Saga에서 메시징에 요구되는 특성은 다음과 같다:

- **이벤트 영속성:** 소비자가 일시적으로 중단되더라도 이벤트가 유실되지 않아야 한다.
- **순서 보장:** 같은 주문에 대한 이벤트는 발행 순서대로 처리되어야 보상 흐름이 꼬이지 않는다.
- **소비자 그룹:** 서비스 인스턴스가 여러 개일 때 같은 이벤트를 중복 처리하지 않아야 한다.

## 후보군

### 1. Apache Kafka

분산 로그 기반 메시징 시스템. 이벤트를 디스크에 영속하고, 파티션 단위로 순서를 보장한다.

- **장점:** 이벤트가 로그로 영속되어 소비자 중단 시에도 유실되지 않는다. 파티션 키(예: orderId)로 같은 주문의 이벤트를 동일 파티션에 모아 순서를 보장한다. 컨슈머 그룹으로 인스턴스 간 중복 처리를 방지한다.
- **단점:** 브로커 클러스터 운영 부담이 있다. 단순한 작업 큐 용도로는 과도할 수 있다.

### 2. RabbitMQ

AMQP 기반 메시지 브로커. 큐에 메시지를 넣고 소비자가 꺼내가는 전통적인 메시지 큐 모델이다.

- **장점:** 라우팅이 유연하고 설정이 직관적이다. 요청-응답 패턴 지원이 자연스럽다.
- **단점:** 기본 동작이 메시지 소비 후 삭제이므로, 이벤트 재처리나 소비자 추가 시 같은 이벤트를 다시 읽을 수 없다. 순서 보장을 위해 단일 큐 + 단일 소비자로 제한하면 처리량이 떨어진다.

### 3. Amazon SQS

AWS 관리형 메시지 큐. 인프라 운영 부담 없이 사용할 수 있다.

- **장점:** 완전 관리형이라 운영 부담이 없다. FIFO 큐로 순서 보장이 가능하다.
- **단점:** AWS에 종속된다. FIFO 큐는 처리량 제한(초당 300건, 배치 시 3,000건)이 있다. 이벤트 영속성 측면에서 소비 후 삭제가 기본이라 RabbitMQ와 같은 한계를 가진다.

## 결정

**Apache Kafka를 선택한다.**

## 근거

Choreography Saga에서 가장 중요한 요구사항은 이벤트 영속성과 순서 보장이다. 보상 이벤트가 유실되면 부분 커밋 불일치가 영구적으로 남고, 순서가 뒤바뀌면 보상 흐름이 꼬인다.

RabbitMQ와 SQS는 메시지 소비 후 삭제가 기본 동작이라, 이벤트 재처리나 장애 복구 시 같은 이벤트를 다시 읽을 수 없다. Kafka는 이벤트를 로그로 영속하므로 소비자가 오프셋을 되돌려 재처리할 수 있고, 파티션 키로 같은 주문의 이벤트 순서를 보장한다.

브로커 운영 부담이라는 트레이드오프가 있지만, 이 프로젝트에서는 Testcontainers로 Kafka 브로커를 띄워 테스트하므로 운영 환경의 부담은 현재 범위 밖이다.
